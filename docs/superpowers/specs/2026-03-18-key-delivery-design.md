# Key Delivery & Opaque Key Support Design Spec

## 1. Overview

Extend crypto-signer to support **key delivery** вЂ” allowing business processes to retrieve decrypted private keys via IPC for use with third-party SDKs that embed signing internally.

### Problem Statement

Many crypto SDKs (Lighter, Hyperliquid, Polymarket) require the private key to be passed directly to the SDK, which handles signing internally. crypto-signer's current "dumb signer" model (`sign_transaction` / `sign_message`) cannot serve these use cases because the signing step cannot be extracted from the SDK.

Additionally, some of these SDKs use non-EVM key formats (e.g., Lighter API keys generated by `lighter.create_api_key()`), which are not secp256k1 and cannot be stored in the current keystore.

### Goals

1. Allow business processes to retrieve decrypted keys by name via IPC (`get_key`)
2. Support storing arbitrary secret bytes (opaque keys) alongside typed EVM keys
3. Auto-detect key type on import, with graceful fallback
4. Provide a zero-code-change migration path via CLI `exec` command
5. Maintain all existing disk encryption and access control guarantees

### Non-Goals

1. SDK-specific adapters or signing proxies вЂ” out of scope
2. Storing non-secret metadata (key_index, account_index) вЂ” stays in project config
3. Changing the existing `sign_transaction` / `sign_message` flow вЂ” unchanged

## 2. Keystore Changes

### New `opaque` Key Type

The `keys` array in `keystore.json` gains support for `key_type: "opaque"`:

```json
{
  "version": 1,
  "kdf": "argon2id",
  "kdf_params": { "memory_cost": 65536, "time_cost": 3, "parallelism": 4 },
  "keys": [
    {
      "name": "my-evm",
      "key_type": "secp256k1",
      "address": "0xAbC...1234",
      "cipher": "aes-256-gcm",
      "salt": "<base64>", "encrypted_key": "<base64>", "iv": "<base64>", "tag": "<base64>"
    },
    {
      "name": "lighter-api",
      "key_type": "opaque",
      "address": null,
      "cipher": "aes-256-gcm",
      "salt": "<base64>", "encrypted_key": "<base64>", "iv": "<base64>", "tag": "<base64>"
    }
  ]
}
```

The keystore `version` remains `1` вЂ” adding `opaque` is backwards-compatible (old keys unchanged, new field value in existing schema).

### Auto-Detection on Import

When `crypto-signer add --name <name> --key` is invoked without `--type`:

1. User enters private key via `getpass` (no echo)
2. Attempt to parse as secp256k1 and derive EVM address
   - Success: store as `key_type="secp256k1"`, `address="0x..."`
   - Failure: store as `key_type="opaque"`, `address=null`, emit warning: "Cannot derive address; storing as opaque key"
3. Encrypt and write to keystore

If `--type opaque` is explicitly provided, skip auto-detection and store as opaque directly.

### Unlock Verification

- `secp256k1` keys: decrypt, derive address, compare with stored address. Mismatch raises `InvalidPasswordError`.
- `opaque` keys: decrypt only. AES-256-GCM authentication tag guarantees integrity (tampering causes decryption failure).

### Key Selection

Existing `sign_transaction` / `sign_message` methods continue to select keys by `chain` type (unchanged).

The new `get_key` method selects keys by `name`. This supports multiple keys of the same type (e.g., two Lighter API keys for different accounts).

## 3. IPC Protocol Changes

### New Method: `get_key`

**Request:**
```json
{
  "version": 1,
  "id": "req-001",
  "method": "get_key",
  "params": { "name": "lighter-api" }
}
```

**Response (success):**
```json
{
  "id": "req-001",
  "result": {
    "name": "lighter-api",
    "key_type": "opaque",
    "key": "<base64-encoded-decrypted-key>",
    "address": null
  }
}
```

**Behavior:**

- Requires UNLOCKED state (else `SignerLockedError`)
- Subject to existing rate limit (shares the per-minute signature quota)
- Unknown name returns `KeyNotFoundError` (code 1010)
- `key` field is base64-encoded (consistent with existing binary data convention)
- `address` is `"0x..."` for secp256k1 keys, `null` for opaque keys
- Each call is logged at INFO level (key name only, never key content)

### New Error Code

| Code | Error Type | Description |
|------|------------|-------------|
| 1010 | KeyNotFoundError | Requested key name does not exist in keystore |

### Existing Methods вЂ” Unchanged

All existing methods (`sign_transaction`, `sign_message`, `sign_typed_data`, `get_address`, `status`, `lock`, `unlock`, `ping`, `shutdown`) remain unchanged.

## 4. Client Library Changes

### New Method: `get_key`

```python
from crypto_signer import SignerClient

client = SignerClient()

# New: retrieve decrypted key by name
key = client.get_key("lighter-api")    # returns str (decoded key)
key = client.get_key("my-evm")         # works for any key type

# Existing: unchanged
client.evm.sign_transaction(tx)
client.evm.get_address()
```

`get_key` returns the key as a string (base64-decoded). The caller passes it to the relevant SDK.

### Usage Example (copytrade)

```python
# Before: private key in .env
signer = lighter.SignerClient(
    api_private_keys={config.lighter_api_key_index: config.lighter_api_private_key},
    account_index=config.lighter_account_index,
    url=config.lighter_base_url,
)

# After: private key from crypto-signer
from crypto_signer import SignerClient
key = SignerClient().get_key("lighter-api")

signer = lighter.SignerClient(
    api_private_keys={config.lighter_api_key_index: key},
    account_index=config.lighter_account_index,
    url=config.lighter_base_url,
)
```

The project `.env` changes from:
```env
LIGHTER_API_PRIVATE_KEY=a1b2c3d4e5f6...    # SECRET вЂ” removed
LIGHTER_API_KEY_INDEX=3                      # non-secret, stays
LIGHTER_ACCOUNT_INDEX=0                      # non-secret, stays
```
To:
```env
LIGHTER_API_KEY_NAME=lighter-api             # reference to crypto-signer key
LIGHTER_API_KEY_INDEX=3
LIGHTER_ACCOUNT_INDEX=0
```

## 5. CLI Changes

### `add` Command Update

```bash
# Auto-detect (recommended)
crypto-signer add --name my-evm --key
crypto-signer add --name lighter-api --key

# Explicit type (optional)
crypto-signer add --name some-key --type opaque --key

# Mnemonic import (secp256k1 only, unchanged)
crypto-signer add --name my-evm --mnemonic
```

`--type` becomes optional. When omitted, auto-detection runs. When `--type opaque` is given, auto-detection is skipped.

### `list` Command Update


### New `exec` Command

```bash
crypto-signer exec \
  --inject lighter-api=LIGHTER_API_PRIVATE_KEY \
  --inject my-evm=ETH_PRIVATE_KEY \
  -- python main.py
```

**Behavior:**

1. Connect to signer daemon (must be UNLOCKED)
2. For each `--inject name=ENV_VAR`, call `get_key(name)` to retrieve decrypted key
3. Set retrieved keys as environment variables for the child process
4. Spawn child process with the specified command (`python main.py`)
5. Wait for child process to exit, return its exit code
6. If any `get_key` fails (name not found, signer locked), abort immediately without starting child process

**Notes:**

- Keys exist only in the child process environment, never written to disk
- Environment variables are destroyed when the child process exits
- The `exec` process itself does not retain the keys after passing them to the child

## 6. Security Considerations

### Security Boundary Change

| Aspect | Existing `sign_transaction` | New `get_key` |
|--------|---------------------------|---------------|
| Key leaves signer process | No | Yes |
| Disk encryption | Preserved | Preserved |
| Password unlock required | Yes | Yes |
| IPC transport protection | Socket perms + peer UID | Same |
| Root attacker surface | Signer process only | Any process holding key |

**Assessment:** Disk security is unchanged. Runtime attack surface expands from 1 process to N processes. For VPS hot wallet scenarios, this is an acceptable trade-off вЂ” if an attacker has root, single-process isolation was never a reliable defense.

### `get_key` Security Measures

- **Rate limited**: Shares per-minute quota with signing requests
- **Requires UNLOCKED state**: Same precondition as signing
- **Audit logging**: Every `get_key` call logged at INFO level (key name only, never key content)
- **IPC permissions unchanged**: Socket file 0600 + peer UID verification (Linux/macOS)

### `exec` Security Notes

- Keys exist in child process environment variables; on Linux, root can read via `/proc/pid/environ`
- Still better than `.env` files: environment variables are destroyed on process exit, `.env` persists on disk
- Documentation must state: `get_key` IPC is preferred over `exec` injection; `exec` is a convenience trade-off

### Documentation Update

`SECURITY.md` must be updated with:

> The `get_key` method delivers decrypted keys to the calling process. Runtime security of the delivered key is the caller's responsibility. Callers should clear keys from memory when no longer needed. For maximum security, prefer `sign_transaction` over `get_key` when the signing model allows it.

## 7. Error Model Update

```
SignerError (base)
+-- ...existing errors...
+-- KeyNotFoundError         # code 1010: requested key name not in keystore
```

Error messages for `get_key` must never contain key content. The key name may appear in error messages.
